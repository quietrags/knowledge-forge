## The Core Insight

Each learning mode isn't a single interface â€” it's a **workspace with multiple specialized panels**. The platform must:

1. Provide mode-specific panel layouts
2. Allow panels to be shown/hidden/rearranged
3. Connect everything through a persistent meta-layer
4. Enable cross-mode navigation without losing context

---

## Mode Interface Requirements

### BUILD Mode Interfaces

From the constructivist pedagogy, /build needs **5 distinct interface areas**:

| Interface | Purpose | Content |
|-----------|---------|---------|
| **Construction Chronicle** | Running narrative of learning journey | Emphasized insights, breakthrough moments, "aha" points, struggle-recovery arcs. Not just stats â€” the *story* of construction. |
| **Anchor Constellation** | Visual map of prior knowledge being used | Which anchors are active, which bridges are being built, connection strengths |
| **Code Workshop** | Interactive code-based learning | Scaffolds to complete, code to examine/debug, learner's working code, execution output |
| **Live Construction State** | Current progress dashboard | Active SLO, round #, scaffold level, mode (normal/surrender/code), pattern observations |
| **Curiosity Radar** | Discovery and connection suggestions | Neighboring ideas, "you might be curious about...", research seeds, topic connections |

## Recommendation

**Option A (IDE-Style)** provides the best balance:

1. **Persistent Knowledge Galaxy** in sidebar â€” always visible, always navigable
2. **Primary Workspace** that adapts per mode with the most important interface
3. **Secondary Panels** as tabs that can be switched/shown/hidden
4. **Persistent Chat** at bottom for conversation
5. **Mode Switcher** in header that reconfigures the entire layout

This matches how developers work (familiar patterns) while supporting the complexity of multi-panel, multi-mode learning.
